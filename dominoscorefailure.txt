What the board actually is
The layout only works if 5/5 is the opening spinner (first tile). The engine only allows a bottom branch off the spinner, not off a 5/5 in the middle of the chain:

0/6 ← 6/3 ← 3/3 ← 3/5 ← [5/5 spinner] → 5/6 → 6/4 → 4/5
                              |
                             5/2 → 2/2 → 2/4 → 4/0
At the snapshot in the file, open ends are 0 (left), 5 (right), 0 (bottom) — a valid 5-point position, not 15.

Why it feels wrong
Your dad likely expected something closer to even because:

You opened and got the first 10 — fair start.
The CPU then stacked four scoring plays on the bottom arm while your left/right chains were still short.
You only got one mid-game score (the 15 on 6/3).
So the gap is mostly who played the scoring tiles, not the scoreboard adding wrong.

The early 5/2 → 10 is the move that looks suspicious, but it’s legal: with ends 3, 5, and 2, the sum is 10. The left arm only needs one tile (3/5); it doesn’t have to be fully built before the bottom arm scores.

What is not wrong in current code (for this board)
Points go to whoever just played — no turn mix-up found.
Spinner rules (ignore 5/5 top/bottom once both sides are filled, then count played arms) behave correctly here.
The final board state (0 + 5 + 0 = 5) scores correctly.
Minor note on the file itself
The “Player Dominoes” / “CPU Dominoes” lists in dominoscorefailure.txt repeat tiles already on the board (e.g. 5/5, 5/2, 4/0). That’s a notation/snapshot issue, not evidence of duplicate tiles in the game.

Bottom line
For this exact board, CPU 45 / You 25 is a real, rules-legal outcome with current code — especially if you opened and the CPU built the bottom branch early. 50 / 25 was not found; 45 / 25 matches closely.
